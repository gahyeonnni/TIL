## [OS] 동기화와 락

### 동기화

- 멀티 스레드를 사용할 때, 스레드는 stack영역만 독자적으로 할당받고 나머지 메모리 영역은 공유
    - heap, data영역에 있는 값에 여러 스레드가 동시에 접근을 하게 되면 ?
        - **경쟁 조건**
            - 여러 프로세스나 스레드가 동시에 같은 데이터를 접근하거나 변경할 때, 어떤 순서로 실행되는지에 따라 예상하지 못한 결과가 발생할 수도 있는 상황
        - → 동시성 문제 발생!
        - → 이러한 문제를 해결하기 위한 것이 동기화
    - 멀티 스레드 환경에서 여러 스레드들이 같은 메모리주소나 값에 접근을 할 때, 서로 작업한 내용들을 일치시키는 것

### **임계 영역**

- 위에서 말한 여러 스레드들이 동시에 접근을 했을 때 문제가 생길 수 있는 부분을 처리하기 위해 한번에 하나의 스레드만 해당 영역에 접근이 가능한 곳
    - 이 임계 영역부분을 통해서 스레드 동기화 문제를 해결할 수 있기 때문에, 이 접근 권한을 주는 것이 매우 중요
    - **Mutual exclusion(상호 배제)**
        - 임계 영역에 한 번에 하나의 스레드만 접근하도록 보장하는 원칙

### 락

- 이 임계 영역에 접근할 수 있는 권한 ( = 동시성 문제를 해결하는 핵심적인 방법! )
- 이 락을 가지고 있는 스레드만 해당 임계 영역에 접근을 가능하도록 하여 임계 영역에 대한 독자적인 권한을 주는 동기화 방식
    - 락 동기화 종류
        - **유저 모드 동기화**
            - **critical section**
                - 윈도우에서 지원하는 동기화 기법중 하나. critical section 생성 -> 권한 획득 -> 권한 반납 -> 제거로 이어짐
                - 엄밀히는 **하이브리드** 방식이다. 경쟁이 없을 때는 유저 모드에서 빠르게 처리하고, 경쟁이 발생하면 커널 모드로 전환해 대기한다. 
                - 그래서 커널 오브젝트인 뮤텍스보다 가볍고 빠르지만, 같은 프로세스 내의 스레드끼리만 사용 가능하다.
            - **Interlock**
                - 윈도우에서 지원하는 동기화 기법으로 interlock~ 으로 시작하는 함수들은 함수를 실행하는데 원자적 실행을 보장해 동기화를 보장함
        - **커널 오브젝트 동기**
            - **뮤텍스 ( Mutex Lock )**
                - 값이 signal 과 nonsignal로 이루어져 있고, 뮤텍스를 스레드가 소유하게 되면, nonsignal로 바뀌고 락을 해제하게 되면 다시 signal 상태로 바뀌는 동기화 기법
                - 하나의 주체만 락을 획득
                - 다른 스레드가 락을 점유중이어서 현재 스레드가 락을 획득할 수 없다면, 이 스레드는 대기 상태로 들어가고, 락이 사용 가능해지면 운영체제가 해당 스레드를 깨운다.
                - 이 방식은 대기 중 CPU 자원을 낭비하지 않는다.
            - **세마포어**
                - 뮤텍스와 달리 0이상의 시그널
                - 내부적으로 카운터를 두고, `P(wait)` 연산으로 카운터를 1 감소시키며 임계 영역에 진입하고, `V(signal)` 연산으로 1 증가시키며 빠져나온다. 
                  - 카운터가 0이 되면 이후 진입하려는 스레드는 대기한다.
                - 즉 **동시에 N개의 스레드까지** 임계 영역 진입을 허용하는 "자원 개수 제한" 용도로 적합 (예: DB 커넥션 풀, 동시 접속자 수 제한)
                - 시그널이 0이 될 때까지 해당 영역안에 스레드를 받을 수 있다
                    - 세마포어안에 있는 스레드가 아닌 다른 스레드들도 signal을 조절 가능
                - Event같은 경우 0과 1의 signal밖에 가지지 못하기 때문에 다른 스레드가 signal을 빠르게 보내게 되면 해당 작업은 수행을 하지 못하게 되는 현상이 발생하는데, 세마포어는 signal값을 조절할 수 있기 때문에 이러한 문제를 해결 가능
                    - -> queue처럼 쌓아두고 작업이 가능
                - 이진 세마포어를 활용해 상호 배제를 구현할 수 있다.
                - Q. 뮤텍스 == 세마포어 ?
                    - **뮤텍스**: 락을 건 스레드만 해제할 수 있고, Priority Inheritance를 지원
                    - **세마포어:** 누구나 락을 해제할 수 있고, 순서 조절에 유용하지만 Priority Inheritance 없음.
                        - **Priority Inheritance란?**
                        - 스레드가 CPU 할당을 너무 오래 기다리는 문제를 방지하기 위해, 낮은 우선순위 스레드가 락을 잡고 있으면 잠깐 우선순위를 올려줘서 빨리 끝내게 하는 것.
            - **이벤트**
                - 이벤트같은 경우는 0과 1의 값을 가지며 1일 때 signal 상태가 되며 해당 이벤트를 wait함으로 어디까지 진행되었고, 지금 어디서 시작할 상황이다 라는 것을 알려줄 수 있음
    - **스핀락 (Spinlock)**
        - 락을 얻을 때까지 대기 상태로 들어가지 않고, 반복문을 돌며 계속 락 획득을 시도하는 방식
        - 뮤텍스처럼 잠들었다 깨어나는 컨텍스트 스위칭 비용이 없어서, 임계 영역이 매우 짧을 때 유리하다.
        - 반대로 대기 시간이 길어지면 CPU를 계속 점유하며 낭비하므로 불리하다. (뮤텍스 = 블로킹/sleep ↔ 스핀락 = 논블로킹/busy-wait)
    - 락 사용 시 주의점: 교착 상태(Deadlock)
        - 두 개 이상의 스레드가 서로가 가진 락을 기다리며 무한히 대기하는 상태
        - 동기화를 위해 락을 과도하게 사용하거나 순서를 잘못 설계하면 시스템이 완전히 멈출 수 있으므로 주의해야 함.
        - **교착 상태의 4가지 필요조건** (4가지가 모두 성립할 때만 발생 → 하나만 깨도 예방 가능)
            1. **상호 배제(Mutual Exclusion)**: 자원을 한 번에 하나의 스레드만 사용
            2. **점유와 대기(Hold and Wait)**: 자원을 가진 채로 다른 자원을 추가로 기다림
            3. **비선점(No Preemption)**: 다른 스레드의 자원을 강제로 빼앗을 수 없음
            4. **순환 대기(Circular Wait)**: 스레드들이 원형으로 서로의 자원을 기다림

### 모니터 (Monitor)

- 락 + 조건 변수를 묶어 객체 수준에서 상호 배제를 보장하는 고수준 동기화 도구
    - 임계 영역을 직접 락으로 감싸는 대신, 공유 자원에 대한 접근을 모니터 안의 메서드로만 가능하게 하여 한 번에 하나의 스레드만 실행되도록 함
- 자바에서는 모든 객체가 모니터를 하나씩 가지고 있으며, `synchronized`가 이 모니터 락을 사용한다.
    - `wait()` / `notify()` / `notifyAll()` 가 모니터의 조건 변수 역할 → 특정 조건이 충족될 때까지 스레드를 대기시키거나 깨움

### 비관적 락 vs 낙관적 락

- **비관적 락(Pessimistic Lock)**
    - 충돌이 자주 일어날 것이라 가정하고, 데이터에 접근하는 순간 먼저 락을 건다.
    - 뮤텍스, `synchronized`, DB의 `SELECT ... FOR UPDATE` 등. 안전하지만 대기·교착 가능성 존재
- **낙관적 락(Optimistic Lock)**
    - 충돌이 드물 것이라 가정하고 락을 걸지 않은 채 진행한 뒤, 커밋 시점에 버전이나 값이 바뀌었는지 검증하여 충돌 시 재시도/롤백
    - CAS, DB의 version 컬럼 방식. 충돌이 적을 때 성능이 좋음

### **Lock과 Synchronized의 차이**

- Synchronized block은 메서드 내부에 선언 ↔ Lock은 메서드와 분리된 lock(), unlock() 메서드를 가질 수 있다.
- Synchronized block은 공정성을 가질 수 없다
    - 스레드는 lock이 release되어야 lock을 얻을 수 있으며 선호도를 지정할 수 없다.
- Lock API
    - 공정성 상태값을 지정할 수 있어서 오랫동안 기다린 스레드에게 lock을 획득하게 할 수 있다.
- Synchronized block에 들어가려고 하는데 이미 다른 스레드가 해당 객체의 모니터 락을 가지고 있으면 이 스레드는 block 상태로 대기
    - **synchronized의 단점 : 무한 대기**
        - 특정 시간까지만 대기하는 타임아웃 기능❌
        - 중간에 다른 스레드가 강제로 깨우는 인터럽트 기능❌
- 자바의 Lock API는 tryLock()이라는 특별한 메서드가 존재하는데 tryLock()을 사용하면 다른 스레드가 Lock을 가지고 있지 않을 때만 Lock을 얻는다. 락이 이미 사용중이면 그냥 실패로 넘어간다. 즉 기다리지 않는다.
